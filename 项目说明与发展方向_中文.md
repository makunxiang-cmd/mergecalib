# mergecalib 项目审核、发展方向与框架说明（中文，仅供参阅）

> 本文件是给你本人参阅的中文说明，**不纳入** GitHub/CRAN 仓库。仓库内所有文档均已英文化。

## 一、项目审核结论

`mergecalib` 已经是一个**结构完整、设计严谨**的 R 包，而不是半成品。核心评价：

**优点**

- 代码组织清晰：`R/` 按职责拆分（spec / validate / candidates / model / fit / results / print），命名规范，缩进统一。
- 算法设计扎实：候选簇生成 + 集合划分 MILP + 字典序分层优化，思路正确，且对"零样本人群必须被吸收""最终簇样本量 > 0"做了硬约束和事后审计（`audit_merge_fit`）。
- 确定性强：固定排序、单线程、固定随机种子，结果可复现——这对统计生产环境很重要。
- 已有测试、vignette、示例数据、DESCRIPTION/NAMESPACE/LICENSE/NEWS、以及一份免责声明。

**投稿 CRAN 之前的主要差距（我已处理的部分见第三节）**

1. 大量**面向用户的字符串是中文**（报错信息、print/plot 标签、示例 target_name），会触发 CRAN 的 non-ASCII NOTE，也与"英文文档"目标冲突。→ 已全部英文化并改为 ASCII 干净。
2. 缺少 **git 仓库、CI、贡献指南、issue 模板、pkgdown、cran-comments** 等开源/CRAN 标准工程化文件。→ 已补齐。
3. 免责声明只是一段静态中文文本，**没有"用户需同意"的机制**。→ 已实现 CRAN 安全的会话内同意机制。
4. DESCRIPTION 缺 `URL`/`BugReports`/`Language`。→ 已补。
5. 测试用例用中文正则匹配报错信息，英文化后会失效。→ 已同步改为英文断言。

## 二、建议的发展方向（定位：投稿 CRAN）

按优先级：

1. **先让 `R CMD check --as-cran` 干净通过**（0 error / 0 warning，NOTE 尽量清零；首次提交会有一个 "New submission" NOTE 属正常）。这是 CRAN 的硬门槛。
2. **完善导出函数的 `\value` 文档**：CRAN 要求每个导出函数的帮助页都有 `\value`。当前 `man/` 是手写的，部分函数（如 `cell_map`、`merge_plan` 等结果访问器）合并在 `results.Rd` 里，需确认每个 alias 都有 value 描述。
3. **统一文档生成方式**：现在 `man/*.Rd` 是手写的（没有 roxygen 头）。建议二选一：要么完全改用 roxygen2 自动生成（推荐，长期好维护），要么继续手写但每次改签名都同步。
4. **补充测试**：放宽量搜索路径（`relax_targets`/`max_delta`）、`export_merge_results` 往返、`calculate_results` 多维汇总，提升覆盖率与可信度。
5. **CI 跑绿** 后打 tag `v0.1.0`，用 `devtools::release()` 提交 CRAN。
6. **可选增强**（CRAN 之后）：pkgdown 文档站、更多求解器选项、性能基准、对超大省份的候选剪枝策略、并行可选开关（但默认仍保持确定性）。

## 四、你需要在本地做的事

1. **安装 R 与依赖后自检**（沙箱里没有 R，我无法替你运行 `R CMD check`）：
   ```r
   install.packages(c("devtools","roxygen2","testthat","highs"))
   devtools::check()        # 目标 0/0/0
   ```
   若你把 `man/` 改成 roxygen 自动生成，先 `devtools::document()`。
2. **推送到 GitHub**（仓库已建好 remote）：
   ```sh
   cd "<包目录>"            # 即 mergecalib_sourcecode/
   git push -u origin main
   ```
   首次推送会让你做 GitHub 认证（浏览器或 PAT）。详见 `mergecalib_sourcecode/GIT_AND_PUSH.md`。
3. CI 跑绿、check 干净后，打 tag 并用 `devtools::release()` 投 CRAN。

## 五、注意事项 / 风险点

- **手写 man 页**：新增/修改导出函数时，必须手动加 `.Rd` 并在 `NAMESPACE` 导出，否则 check 报 "undocumented"。
- **报错信息与测试断言耦合**：改报错文案要同步改测试。
- **确定性**：不要在默认行为里引入随机或多线程。
- **同意机制**：保持非交互不阻塞、不写盘，否则会破坏 CRAN check。
- **DESCRIPTION 邮箱**（`makunxiangiphone@icloud.com`）必须能收信，CRAN 会发确认邮件。
